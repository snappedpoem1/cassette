Three approaches to the full sweep

Same destination — different engineering paths. My recommendation is highlighted.

A

Layered CSS system — my recommendation

Extend the existing --mood-* CSS variable system to cover every morph surface. Add a ::before backdrop blob layer on .app-shell, push grain as a pseudo-element, wire all glow/border colors to derived mood vars. The layout.svelte color sampler already does the heavy lifting — we just widen what it feeds.

Pros
  • Builds on working code
  • CSS transitions = free 60fps
  • low_motion already respected
  • No new JS complexity
Cons
  • Backdrop blobs need careful perf tuning
  • Grain overlay is a pseudo-element trick
B

Canvas backdrop renderer

A dedicated <canvas> behind the app shell renders animated color blobs using the sampled track color. More control over the animation, true "morphing" blobs, can react to audio amplitude. More complex to build and maintain.

Pros
  • Most expressive motion
  • Could react to actual audio
Cons
  • New JS animation loop
  • Harder to respect low_motion
  • Overkill for the backdrop layer
C

Hybrid — CSS system + one JS enhancement

Do the CSS variable expansion (approach A) for everything except the backdrop blobs, where we use a lightweight JS-driven CSS animation that morphs between two blob positions using @keyframes with dynamic inline styles. Best of both: CSS does the heavy lifting, JS only touches the backdrop.

Pros
  • Backdrop actually morphs shape
  • Still CSS-native for transitions
  • Manageable complexity
Cons
  • Slightly more code than pure CSS
My recommendation: A (with blob animation from C if you want it)

The existing color sampler in +layout.svelte already produces --mood-accent-rgb per track. Widening that to drive the whole visual system is low-risk, fast to ship, and produces the morph effect you're after. We can always layer the animated blobs on top later.
